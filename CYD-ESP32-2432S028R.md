# CYD 2.8" — ESP32-2432S028R — firmware notes (LovyanGFX)

Hard-won reference for the 2.8" "Cheap Yellow Display" (ESP32-2432S028R, 2.8"
ILI9341, resistive touch, **no PSRAM**) as wired up for **Overhead** with
**LovyanGFX** (`cyd28_ili9341`). Covers the hardware + display-orientation + touch
fixes **and** the runtime quirks (screenshot colour, no-PSRAM budget, OTA, WiFi
watchdog) that each cost real debugging time — read before changing display,
screenshot, OTA, or WiFi code. For the 4" CYD (ST7796) and the CrowPanel, see
`hal/Board.h`; for the autonomous flash/screenshot/control loop see `PIO_DEBUG.md`.

## The board
ESP32-WROOM-32 (4 MB flash, no PSRAM) + 2.8" 240×320 **ILI9341** LCD + **XPT2046**
resistive touch, SD slot, LDR, RGB LED. Two revisions: single-USB (micro-USB) and
dual-USB (USB-C + micro). The dual-USB ILI9341 is wired with reversed R/B polarity.

## Pin reference (the ones we use)
```
Display (ILI9341, HSPI):           Touch (XPT2046, its OWN VSPI bus):
  MISO  12   MOSI 13   SCLK 14       T_CLK 25   T_DIN(MOSI) 32   T_OUT(MISO) 39
  CS    15   DC    2   RST  -1        T_CS  33   T_IRQ 36
  BL    21  (backlight, active HIGH)
SD (HSPI shared): CS 5 / SCK 18 / MOSI 23 / MISO 19
I2C header (free): GPIO 27 (SDA), 22 (SCL)   RGB LED (active LOW): 4/16/17
```
Backlight PWM: use a LOW frequency (≤5 kHz) — the on-board MOSFET can't switch off
fast enough above ~10 kHz, so high-freq PWM does nothing. Drive **GPIO21 with the
Arduino LEDC API directly** (`ledcSetup`/`ledcAttachPin`/`ledcWrite`, channel 7):
LovyanGFX's `Light_PWM` did **not** drive this unit's backlight (screen stuck at one
brightness) — don't go back to it.

## Display orientation — THE fix (Overhead, LovyanGFX)
This panel's ILI9341 is mounted so the chip's **MV=1 "landscape" modes** (LovyanGFX
*odd* rotations 1/3) render **90°-rotated** and only differ by 180° mirror bits —
so no odd rotation is ever upright. The orientation that visually appears as
correct landscape is the chip's **MV=0** mode, and on this CYD the column range
extends so a full 320-wide layout still fits. So we drive the panel as
**landscape-native 320×240 at MV=0**, NOT via a swapping rotation:

```cpp
// LGFX panel config (hal/LGFX_Config.h, BOARD_CYD28_ILI9341):
cfg.panel_width  = 320;  cfg.panel_height  = 240;   // landscape-native
cfg.memory_width = 320;  cfg.memory_height = 240;
cfg.bus_shared   = false;                           // touch is on its own VSPI bus
cfg.rgb_order    = (CYD_PANEL_RGB_ORDER != 0);      // dual-USB units: drop BGR -> 1
// then:  setRotation(DISPLAY_DEFAULT_ROTATION);
```
`DISPLAY_DEFAULT_ROTATION = 6` (MV=0 + horizontal mirror) is upright + un-mirrored
on the tested unit. The MV=0 family (even rotations, all report 320×240):
`0`=upright+mirrored, `2`=upside-down+un-mirrored, `4`=upside-down+mirrored,
`6`=upright+un-mirrored. Use `2` to flip 180°.

**Colour:** dual-USB variants have reversed R/B wiring; without RGB order the warm
theme colours (yellow/orange) render as cyan/blue. Set `CYD_PANEL_RGB_ORDER 1`
(maps to dropping the MADCTL BGR bit). Single-USB units that look right: set 0.

This mirrors yoRadio's solution for the same board
(`../Voxalon/yoradio-pio` displayILI9341.cpp: `DSP_PANEL_SWAP_XY` + `DSP_PANEL_RGB`).
What did NOT work: `setRotation(1)`/`(3)` (MV=1 → 90° off); MADCTL hacks `0x68`/`0xA8`
(only flip 180°); a setAddrWindow x/y transpose (gave 90° + mirror).

## Touch — matching the display orientation
LovyanGFX's `calibrateTouch()` draws its corner targets using the panel's
coordinate system. Because rotation 6 includes a **horizontal flip**, the targets
appear at physically-flipped positions, so the learned calibration comes out
**180° point-reflected** (tapping top-right hits the bottom-left target). Fix: in
`hal/Board.h` for this board, invert both calibrated axes; `hal/Touch::read()`
applies it:
```cpp
#define TOUCH_INVERT_X 1
#define TOUCH_INVERT_Y 1
// in Touch::read(): x = width()-1-x; y = height()-1-y;  (guarded by the flags)
```
Other CYD quirks worth keeping:
- The XPT2046 lives on a **separate VSPI bus** (CLK 25 / MOSI 32 / MISO 39 / CS 33),
  initialised independently of the display's HSPI.
- The library's default `Z_THRESHOLD` (400) is too high for many CYD panels — light
  taps don't register; LovyanGFX's XPT2046 driver is fine in practice here.
- We **tag the saved calibration with the active rotation** (`hal/Touch.cpp`), so
  changing `DISPLAY_DEFAULT_ROTATION` auto-invalidates it and re-runs the 4-corner
  routine — no stale calibration after an orientation change.

## LittleFS gotcha (cost us a debugging session)
Arduino's `LittleFS.begin()` looks for a partition **labelled `spiffs`** by default.
Our partition table must label the data partition `spiffs` (subtype also `spiffs`).
If it's named anything else the mount fails (`partition "spiffs" could not be
found`) and every settings/cache/TLE write silently fails. See `partitions.csv`.

## Platform / RAM note
This no-PSRAM ESP32 has ~190 KB free heap on **arduino-esp32 2.0.x** but only
~90 KB on **3.x** — too little for the TLS reads (HTTPS bodies came back empty on
3.x). So the CYD envs stay on `espressif32 @ 6.9.0` (2.0.x); only the PSRAM
CrowPanel uses pioarduino 3.x. See `platformio.ini`.

## JPEG screenshot — the colour quirk (subtle!)
The `/api/screen.jpg` debug endpoint reads the panel back and JPEG-encodes it
(bitbank2 **JPEGENC**) on the UI thread (`Display::serviceShot`). The pixel mapping
is **panel-specific, derived empirically by sampling known theme colours** — expect
to re-derive it on a different panel:

1. The 16-bit value read back must be **byte-swapped**: `c = (c >> 8) | (c << 8)`.
2. After the swap the channels are **not** the usual RGB565 order — they decode as:
   - **hi 5 bits = Blue**  → `((c >> 11) & 0x1f) << 3`
   - **mid 6 bits = Red**  → `((c >> 5)  & 0x3f) << 2`
   - **lo 5 bits = Green** → `( c        & 0x1f) << 3`
3. JPEGENC then wants the RGB888 bytes written in **B, G, R** order.

Symptoms if wrong: blue/orange swapped, "crazy/false" colours, or pixelated
garbage. Other constraints: the JPEG buffer is **16 KB malloc'd once at boot** (a
runtime malloc/free of that size fragments the heap below the TLS floor); dense
screens overflow it at medium quality → `serviceShot` retries at low quality, then
returns 503; encoding runs on the **UI/loop task**, never the AsyncTCP task.

## No-PSRAM RAM budget (the binding constraint)
- **TLS needs a ~40 KB contiguous block.** `NetClient` skips an HTTPS fetch when the
  largest free block < 42 KB (avoids an OOM heap-corruption crash); providers serve
  stale and retry. Any big transient allocation can knock providers offline.
- **The TLS floor can't be lowered on this platform.** mbedtls' 16 KB RX/TX content
  buffers are baked into the **precompiled** arduino-esp32 (classic `espressif32`)
  libs; `WiFiClientSecure::setBufferSizes()` doesn't exist in 2.0.x. Shrinking them
  needs an IDF/pioarduino source build — so ~40 KB contiguous is the hard TLS cost.
- **HTTP uses one serialized queue.** `NetClient` runs a single core-0 `NetTask`
  draining one request queue — only **one TLS session at a time**, never concurrent
  (concurrent TLS = multiple 40 KB peaks, worse). Keep it serial.
- **Retained data pins the floor → starvation loop.** Holding String-heavy data
  (e.g. 24 ADS-B contacts) under the floor makes `NetClient` skip the very fetch
  meant to refresh it. Fix: **release stale data to free heap** —
  `AircraftProvider::poll()` clears (and `shrink_to_fit`s) contacts older than 60 s.
- Data is trimmed to fit: TLEs **watchlist-only**, aircraft capped at 24, METARs at
  12. (Lift these only on the PSRAM CrowPanel.) No room for a full-screen sprite
  double-buffer (320×224×2 ≈ 143 KB) — flicker is fought with `startWrite()`
  batching + redraw-on-change instead.
- **Monitor pressure via `/api/status`:** `heapBlk` (current largest free block),
  `heapBlkMin` (**low-water** — the key number; below ~42 K = HTTPS starved),
  `httpsSkip` (fetches skipped at the floor; climbing = chronic pressure).

## OTA (ElegantOTA v3 over WiFi)
The single **AsyncTCP** task is the bottleneck. When it's busy (a `/remote` browser
open, a screenshot mid-transfer, or boot-time HTTPS fan-out) the OTA upload returns
**`upload=100`** then **`400`**. Recovery: let the device **boot-settle ~20 s**, or
**two-tap Reboot** on Health (or serial reset, below), then retry → `upload=200`.
Boot fires ~12 HTTPS jobs serially; flashing during that window is what most often
flakes — wait for uptime > ~20 s.

## WiFi drops + watchdog
The radio occasionally **drops and does not auto-recover**, killing OTA and the
debug API (status times out, `upload=000`). A WiFi watchdog in `loop()` nudges
`WiFi.reconnect()` after ~8 s down and **reboots after ~90 s** down (suppressed in
offline field mode); `WiFi.setAutoReconnect(true)` is also set. **Manual recovery
when wedged:** hard-reset over USB serial (esptool asserts DTR/RTS;
`--after hard_reset` boots the app):
```
~/.platformio/penv/Scripts/python.exe -m esptool --port COM5 --before default_reset --after hard_reset flash_id
```
CH340 is on **COM5**. Wait ~18 s for boot + WiFi rejoin, then OTA-flash normally.

## Settings
`Settings::backfillDefaults()` re-adds any missing keys on every load. This exists
because a stale `settings.json` + the web form's "save all" once blanked keys to
0/false (broke alerts/backlight/timeouts). Never let a form write keys it didn't load.

## Toolchain
- PlatformIO isn't on PATH:
  `~/.platformio/penv/Scripts/platformio.exe run -e cyd28_ili9341`.
- The S3 CrowPanel env uses the **pioarduino** platform (arduino-esp32 3.x); its
